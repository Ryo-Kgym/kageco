# Junie コーディング規約

このドキュメントは、当リポジトリにおける Junie 使用時のコーディング基準とプロジェクトルールを定義します。

## 1. プロジェクト構成

- 既存のフォルダ構成を厳守すること。
- 新しいモジュールは、責務に応じた適切なディレクトリに配置すること。
- 推論を厳禁とする。できないこと・わからないことは潔く回答すること。

## 2. コーディング基準

- 使用言語は TypeScript とする。
- 型定義は厳格に行い、`any` の使用は禁止とする。（やむを得ない場合は理由をコメントすること）
- 可能な限り `const` を使用し、`let` は必要な場合のみ使用すること。
- Reactでは関数コンポーネントのみを使用すること。（クラスコンポーネントは禁止）
- コンポーネント名およびカスタムフック名はパスカルケース（例：UserProfile）とすること。
- カスタムフックは `use` を接頭辞に付与すること（例：useFetchData）。
- JsDocは日本語で記述すること。
- クライアントコンポーネントにおいては、最上位のものだけでよい。例えば、A が Bを呼び出す時は、Aのみでよい。
- リネームしたファイルで不要になったファイルは削除すること。

## 3. コードスタイル

- ファイル名は kebab-case（例：user-profile.tsx）とすること。既存のファイルに揃える必要はなく、手を加える場合は、変更してもよい。
- 変数名、関数名は camelCase とすること。
- 型名、インターフェース名、コンポーネント名は PascalCase とすること。
- 文字列は原則シングルクオート `'` を使用すること。

## 4. テスト方針

- 挙動に影響するコード変更時は、必ずテストを実施すること。
- 使用するテストフレームワークは、React Testing Library、Vitestなど標準的なものとすること。
- 重要なロジックにはテストカバレッジを確保すること。
- vitest.testのnameは、ケース名と期待値を日本語で記載すること。例：「〇〇の場合、XXが返ること」「〇〇の場合、エラーが発生すること」

## 5. ビルド・Lint方針

- コード変更後はプロジェクトが正常にビルドできることを確認すること。
- biomeのルールに則ること。

## 6. AI生成コードの取り扱い

- コンパイルエラーがない状態で完了すること。
- Junieが生成したコードは必ず人間によるレビューを実施すること。
- セキュリティ、パフォーマンス、可読性の観点から検証すること。
- 生成コードも本ガイドラインに準拠すること。

## 7. 責務ごとの分割

- 文脈が異なるコード（例：API通信処理と画面描画処理）は別モジュール・別ファイルに分割すること。
- ファイルは単一責務を基本とし、巨大なファイルにならないよう注意すること。

## 8. Gateway / Repository

- サーバーサイドにおいて、永続化を担うinterfaceをgatewayという。
- gatewayの実装をrepositoryという。
- gateway.ts については、JsDocを記述すること。
- repository.ts については、JsDocを記述しない。また、引数、戻り値は、宣言しなくてもコンパイルが通るので、宣言しない。
